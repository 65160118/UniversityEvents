## เนื้อหา
### 1. บทนำ (INTRODUCTION)

#### 1.1 วัตถุประสงค์ (Purpose)
เอกสารนี้จัดทำขึ้นเพื่อระบุความต้องการของซอฟต์แวร์ ระบบจัดการกิจกรรมมหาวิทยาลัย (Event Management System) ซึ่งเป็นระบบบนเว็บที่ช่วยให้ นักศึกษา อาจารย์ และผู้ดูแลระบบ สามารถ
- สร้างกิจกรรม
- สมัครเข้าร่วมกิจกรรม
- เช็กชื่อผู้เข้าร่วม
- ดูรายงานและสถิติของกิจกรรม

เอกสารนี้ทำหน้าที่เป็นข้อตกลงระหว่างทีมพัฒนา ผู้สอน และผู้มีส่วนได้ส่วนเสีย

#### 1.2 ขอบเขต (Scope)
ระบบ Event Management System ประกอบด้วยฟังก์ชันหลักดังนี้
- การสมัครสมาชิกและเข้าสู่ระบบ
- การสร้าง แก้ไข และอนุมัติกิจกรรม
- การแสดงรายการกิจกรรม
- การสมัครเข้าร่วมกิจกรรม
- การจำกัดจำนวนผู้เข้าร่วม
- การเช็กชื่อและบันทึกการเข้าร่วม
- การดูประวัติการเข้าร่วมกิจกรรม
- ระบบรายงานสรุปผลและ Dashboard
- การประเมินความพึงพอใจหลังจบกิจกรรม
- การค้นหาและกรองกิจกรรม

#### 1.3 ภาพรวมเอกสาร (Document Overview)
เอกสารนี้แบ่งออกเป็น 6 หัวข้อหลัก
- บทนำ: ให้ข้อมูลพื้นฐานเกี่ยวกับโครงการ
- คำอธิบายโดยรวม: อธิบายบริบทและลักษณะทั่วไป
- ความต้องการเฉพาะ: ระบุรายละเอียดความต้องการ
- อินเทอร์เฟซภายนอก: กำหนดการเชื่อมต่อระบบ
- ฟีเจอร์ระบบ: อธิบายฟีเจอร์แต่ละรายการ
- ความต้องการอื่นๆ: กฎและข้อกำหนดเพิ่มเติม

#### 1.4 เอกสารอ้างอิง (References)
เอกสารและมาตรฐานที่ใช้อ้างอิงในการจัดทำเอกสารความต้องการระบบจัดการกิจกรรมมหาวิทยาลัย มีดังนี้
- IEEE Std 830-1998 – IEEE Recommended Practice for Software Requirements Specifications
- IEEE Std 29148-2018 – Systems and Software Engineering — Life Cycle Processes — Requirements Engineering
- SWEBOK (Software Engineering Body of Knowledge) Version 3.0
- WCAG 2.1 – Web Content Accessibility Guidelines https://www.w3.org/WAI/WCAG21/quickref/
- พระราชบัญญัติคุ้มครองข้อมูลส่วนบุคคล (PDPA) ประเทศไทย https://www.pdpc.go.th/
- OWASP Top 10 – Web Application Security Risks
- เอกสารอ้างอิงด้านการพัฒนาระบบเว็บและฐานข้อมูลสำหรับสถาบันการศึกษา

หมายเหตุ: เอกสารอ้างอิงถูกเลือกให้สอดคล้องกับระบบสารสนเทศเพื่อการศึกษาและการจัดการกิจกรรมภายในมหาวิทยาลัย

#### 1.5 ข้อตกลงเอกสาร (Document Conventions)
เอกสารนี้ใช้รูปแบบและสัญลักษณ์มาตรฐานเพื่อให้เข้าใจตรงกัน โดยมีข้อตกลงดังต่อไปนี้
- REQ-XXX : ความต้องการเชิงหน้าที่ (Functional Requirements)
- NFREQ-XXX : ความต้องการที่ไม่ใช่เชิงหน้าที่ (Non-Functional Requirements)
- BR-XXX : กฎทางธุรกิจของระบบ (Business Rules)
- DR-XXX : ความต้องการด้านข้อมูล (Data Requirements)
- LC-XXX : ความต้องการด้านกฎหมายและข้อบังคับ (Legal / Compliance Requirements)
- [TBD] : รายการที่ยังไม่ได้กำหนดรายละเอียด (To Be Determined)
- [Optional] : ฟังก์ชันหรือความสามารถเพิ่มเติมที่อาจรวมในอนาคต
- การอธิบายลำดับขั้นตอนการทำงาน (Process Flow) ใช้คำอธิบายเชิงข้อความหรือแผนภาพ
- ความต้องการด้านความปลอดภัย (Security Requirements) จะถูกระบุแยกอย่างชัดเจนในเอกสาร

### 2. คำอธิบายโดยรวม (OVERALL DESCRIPTION)

#### 2.1 มุมมองผลิตภัณฑ์ (Product Perspective)
ระบบจัดการกิจกรรมมหาวิทยาลัยเป็น Web Application ทำงานแบบ Client–Server รองรับการใช้งานผ่าน
- คอมพิวเตอร์
- แท็บเล็ต
- สมาร์ตโฟน

#### 2.2 ฟังก์ชันผลิตภัณฑ์ (Product Functions)
ระบบจัดการกิจกรรมมหาวิทยาลัยจะมีฟังก์ชันหลักดังต่อไปนี้
1. ระบบรองรับการสมัครผู้ใช้ใหม่ โดยนักศึกษาและอาจารย์สามารถสมัครบัญชีด้วยอีเมลและรหัสผ่าน
2. ระบบรองรับการเข้าสู่ระบบของผู้ใช้ที่ลงทะเบียนแล้ว เพื่อใช้งานฟังก์ชันต่าง ๆ ของระบบ
3. ระบบอนุญาตให้ผู้จัดกิจกรรมหรืออาจารย์สามารถสร้างกิจกรรมใหม่ เพื่อเปิดให้ผู้ใช้รายอื่นสมัครเข้าร่วม
4. ระบบอนุญาตให้ผู้จัดกิจกรรมแก้ไขรายละเอียดของกิจกรรม เพื่อให้ข้อมูลเป็นปัจจุบัน
5. ระบบแสดงรายการกิจกรรมทั้งหมดในระบบให้ผู้ใช้สามารถดูรายละเอียดได้
6. ระบบอนุญาตให้นักศึกษาสมัครเข้าร่วมกิจกรรมที่สนใจ
7. ระบบจำกัดจำนวนผู้เข้าร่วมกิจกรรม และปิดรับสมัครอัตโนมัติเมื่อจำนวนผู้เข้าร่วมครบตามที่กำหนด
8. ระบบอนุญาตให้ผู้จัดกิจกรรมเช็กชื่อและบันทึกการเข้าร่วมของผู้สมัคร
9. ระบบแสดงประวัติการเข้าร่วมกิจกรรมย้อนหลังของนักศึกษา
10. ระบบแสดงรายงานสรุปผลและแดชบอร์ดสถิติกิจกรรม สำหรับผู้ดูแลระบบหรือผู้จัดกิจกรรม
11. ระบบรองรับกระบวนการอนุมัติหรือปฏิเสธกิจกรรมก่อนเผยแพร่โดยอาจารย์หรือผู้ดูแลระบบ
12. ระบบอนุญาตให้นักศึกษาประเมินความพึงพอใจของกิจกรรมหลังจากเข้าร่วมแล้ว
13. ระบบรองรับการค้นหาและกรองกิจกรรมตามเงื่อนไขที่ผู้ใช้กำหนด

#### 2.3 ประเภทผู้ใช้ (User Classes)
ระบบจัดการกิจกรรมมหาวิทยาลัยแบ่งผู้ใช้งานออกเป็น 3 ประเภทหลัก ดังนี้

1. **นักศึกษา (Student)**  
   นักศึกษาเป็นผู้ใช้หลักของระบบ มีความสามารถดังต่อไปนี้
   - สมัครและเข้าสู่ระบบ
   - ดูรายการและค้นหากิจกรรม
   - สมัครเข้าร่วมกิจกรรม
   - ดูประวัติการเข้าร่วมกิจกรรม
   - ประเมินความพึงพอใจกิจกรรมหลังเข้าร่วม

2. **อาจารย์ / ผู้จัดกิจกรรม (Instructor / Event Organizer)**  
   อาจารย์หรือผู้จัดกิจกรรมมีหน้าที่ในการจัดการกิจกรรม และสามารถ
   - สมัครและเข้าสู่ระบบ
   - สร้างและแก้ไขกิจกรรม
   - เช็กชื่อและบันทึกการเข้าร่วมกิจกรรม
   - ดูรายงานสรุปผลและสถิติกิจกรรม
   - อนุมัติหรือปฏิเสธกิจกรรม (ตามสิทธิ์ที่ได้รับ)

3. **ผู้ดูแลระบบ (Administrator)**  
   ผู้ดูแลระบบเป็นผู้ควบคุมภาพรวมของระบบ มีความสามารถดังต่อไปนี้
   - อนุมัติหรือปฏิเสธกิจกรรมก่อนเผยแพร่
   - ตรวจสอบและจัดการข้อมูลในระบบ
   - ดูภาพรวมระบบผ่านรายงานและแดชบอร์ด

#### 2.4 สภาพแวดล้อมการทำงาน (Operating Environment)
a) **ฮาร์ดแวร์เซิร์ฟเวอร์ (Server Hardware)**
- เซิร์ฟเวอร์แบบ Cloud หรือ On-Premise
- AWS EC2 หรือเครื่องแม่ข่ายของมหาวิทยาลัย
- หน่วยความจำ: อย่างน้อย 4–8 GB RAM
- CPU: 2 vCPU หรือเทียบเท่า
- พื้นที่เก็บข้อมูล: อย่างน้อย 50 GB  
  หมายเหตุ: รองรับการขยายระบบในอนาคตเมื่อจำนวนกิจกรรมและผู้ใช้งานเพิ่มขึ้น

b) **ฮาร์ดแวร์ฝั่งผู้ใช้ (Client Hardware)**
- อุปกรณ์: PC, Laptop, Tablet, Smartphone
- หน่วยความจำขั้นต่ำ: 2 GB RAM
- พื้นที่ว่าง: อย่างน้อย 100 MB
- การเชื่อมต่ออินเทอร์เน็ต: 1 Mbps ขึ้นไป

c) **ระบบปฏิบัติการ (Operating Systems)**
- **เซิร์ฟเวอร์**
  - Ubuntu 20.04 LTS หรือใหม่กว่า
  - Windows Server (ถ้าเป็นเครื่องของมหาวิทยาลัย)
- **ฝั่งผู้ใช้**
  - Windows 10 ขึ้นไป
  - macOS 10.15 ขึ้นไป
  - iOS 14 ขึ้นไป
  - Android 10 ขึ้นไป

d) **เบราว์เซอร์ที่สนับสนุน (Supported Browsers)**
- Google Chrome
- Mozilla Firefox
- Safari
- Microsoft Edge  
  (รองรับเวอร์ชันล่าสุดอย่างน้อย 2 เวอร์ชันย้อนหลัง)

#### 2.5 ข้อจำกัดในการออกแบบและการใช้งาน (Design and Implementation Constraints)
a) **ข้อจำกัดด้านเทคโนโลยี (Technology Constraints)**
- ต้องใช้ React สำหรับ Frontend
- Backend: Node.js / Express
- ฐานข้อมูล: MySQL เท่านั้น

b) **ข้อจำกัดด้านอัตราคนหนึ่ง (Schedule Constraints)**
- ระยะเวลาพัฒนาตามกรอบรายวิชา
- ต้องส่งมอบงานตามกำหนดของอาจารย์ผู้สอน
- การพัฒนาระบบแบ่งเป็นระยะ:
  - วิเคราะห์และออกแบบระบบ
  - พัฒนาและทดสอบ
  - นำเสนอผลงาน

c) **ข้อจำกัดด้านงบประมาณ (Budget Constraints)**
- เป็นโครงการเพื่อการศึกษา
- ใช้ซอฟต์แวร์แบบ Open Source
- ไม่มีค่าใช้จ่ายด้านลิขสิทธิ์ซอฟต์แวร์
- หากใช้ Cloud จะใช้ในระดับ Free Tier หรือค่าใช้จ่ายต่ำ

d) **ข้อจำกัดด้านทีมพัฒนา (Team Constraints)**
- ทีมพัฒนาเป็นนักศึกษาจำนวนจำกัด
- สมาชิกแต่ละคนทำหน้าที่หลายบทบาท
- ไม่มีทีมทดสอบแยกต่างหาก
- การทดสอบระบบทำโดยทีมพัฒนาเอง

#### 2.6 สมมติฐานและการพึ่งพา (Assumptions and Dependencies)
a) **สมมติฐาน (Assumptions)**
- ผู้ใช้มีอุปกรณ์และอินเทอร์เน็ตสำหรับเข้าใช้งานระบบ
- ข้อมูลกิจกรรมถูกกรอกโดยผู้จัดกิจกรรมอย่างถูกต้อง
- ผู้ใช้มีความรู้พื้นฐานในการใช้งานเว็บไซต์
- ระบบถูกใช้งานภายในบริบทของมหาวิทยาลัย

b) **การพึ่งพา (Dependencies)**
- พึ่งพาเครือข่ายอินเทอร์เน็ตของมหาวิทยาลัย
- พึ่งพาเซิร์ฟเวอร์หรือระบบโฮสติ้งที่ใช้ติดตั้งระบบ
- พึ่งพาเว็บเบราว์เซอร์ของผู้ใช้งาน
- พึ่งพาฐานข้อมูลสำหรับจัดเก็บข้อมูลกิจกรรมและผู้ใช้

### 3. ความต้องการเฉพาะ (SPECIFIC REQUIREMENTS)

#### 3.1 ความต้องการเชิงหน้าที่ (FUNCTIONAL REQUIREMENTS)
**REQ-001: การสมัครผู้ใช้ (User Registration)**  
ระบบจะต้องอนุญาตให้ผู้ใช้ใหม่ ซึ่งประกอบด้วยนักศึกษาและอาจารย์ สมัครบัญชีผู้ใช้งานโดยใช้อีเมลและรหัสผ่าน ระบบต้องตรวจสอบความถูกต้องของรูปแบบอีเมล และความเหมาะสมของรหัสผ่านก่อนสร้างบัญชีผู้ใช้
- ลำดับความสำคัญ (Priority): สูง (HIGH)
- สถานะ (Status): อนุมัติแล้ว (Approved)

**REQ-002: การเข้าสู่ระบบ (User Login)**  
ระบบจะต้องอนุญาตให้ผู้ใช้ที่ลงทะเบียนแล้วเข้าสู่ระบบด้วยอีเมลและรหัสผ่าน เพื่อเข้าถึงฟังก์ชันต่าง ๆ ของระบบจัดการกิจกรรม
- ลำดับความสำคัญ (Priority): สูง (HIGH)
- สถานะ (Status): อนุมัติแล้ว (Approved)

**REQ-003: การสร้างกิจกรรม (Create Event)**  
ระบบจะต้องอนุญาตให้ผู้จัดกิจกรรมหรืออาจารย์สามารถสร้างกิจกรรมใหม่ โดยกำหนดข้อมูลกิจกรรม เช่น ชื่อกิจกรรม วันเวลา สถานที่ รายละเอียด และจำนวนผู้เข้าร่วมสูงสุด เพื่อเปิดให้ผู้ใช้รายอื่นสมัครเข้าร่วมกิจกรรมได้
- ลำดับความสำคัญ (Priority): สูง (HIGH)
- สถานะ (Status): อนุมัติแล้ว (Approved)

**REQ-004: การแก้ไขกิจกรรม (Edit Event)**  
ระบบจะต้องอนุญาตให้ผู้จัดกิจกรรมสามารถแก้ไขรายละเอียดของกิจกรรม เพื่อให้ข้อมูลกิจกรรมเป็นปัจจุบันและถูกต้อง
- ลำดับความสำคัญ (Priority): ปานกลาง (MEDIUM)
- สถานะ (Status): อนุมัติแล้ว (Approved)

- **ออกจากระบบ**  
  ผู้ใช้กดออกจากระบบ → ระบบล้าง Session

### 6. ความต้องการอื่น ๆ (OTHER NON-FUNCTIONAL REQUIREMENTS)

#### 6.1 กฎทางธุรกิจ (Business Rules)
- BR-001: นักศึกษาสามารถสมัครกิจกรรมได้เฉพาะกิจกรรมที่เปิดรับสมัคร
- BR-002: นักศึกษาหนึ่งคนสมัครกิจกรรมเดียวกันได้เพียงครั้งเดียว
- BR-003: กิจกรรมต้องได้รับการอนุมัติก่อนเผยแพร่
- BR-004: สามารถประเมินกิจกรรมได้เฉพาะผู้ที่เข้าร่วมจริงเท่านั้น

#### 6.2 ความต้องการด้านข้อมูล (Data Requirements)
- **DR-001 ข้อมูลผู้ใช้**
  - ชื่อ นามสกุล
  - อีเมล
  - ประเภทผู้ใช้
- **DR-002 ข้อมูลกิจกรรม**
  - ชื่อกิจกรรม
  - วันเวลา
  - สถานที่
  - จำนวนผู้เข้าร่วมสูงสุด
- **DR-003 ข้อมูลการเข้าร่วม**
  - รหัสผู้ใช้
  - รหัสกิจกรรม
  - สถานะการเข้าร่วม

#### 6.3 กฎหมายและข้อบังคับ (Legal / Compliance)
- LC-001 ปฏิบัติตามกฎหมาย PDPA ประเทศไทย
- LC-002 ข้อมูลส่วนบุคคลต้องเข้ารหัสในฐานข้อมูล
- LC-003 ผู้ใช้สามารถขอลบข้อมูลส่วนตัวได้
- LC-004 ระบบต้องมีการบันทึก Log การใช้งานเพื่อการตรวจสอบ
